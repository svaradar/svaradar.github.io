Introduction to Powerbuilder

PB101 contd - PB Objects

Is a PB Application Object Oriented?

We touched on PB objects while discussing hierarchy  in the last post. Powerbuilder IDE itself is built in an objected oriented way. By creating controls based on the widgets in PB, you are already creating your own objects. So, in theory your application must be Object-oriented. But not quite. There is a lot of PB applications that are built more in structured programming methodology than OO methodology. So, most of the PB apps are essentially what someone called “Object Wrapped” code.

In PB (like C++) you can build completely object oriented applications as well. To facilitate this, there were some frameworks/Class Libraries developed both by Powersoft and third party. The most popular and important one is PFC - Powerbuilder Foundation Classes. When you build your application based on these, your program is more object oriented than standalone PB apps. But, one could actually code Structural code inside such OO environment as well.

Classes, Objects

In a classic Object Oriented languages like Java, you hear about Classes and Objects. A Class is typically mentioned as a "blue print" for an object. It's equivalent to a Data Type. (In fact, a class is defined as an Abstract Data Type). An object on the other hand, is the actual instance of the class.

In Java for e.g., you define class. In fact, all your source codes are defined inside some classes. You compile these files, to generate Class files. At run time, these classes are run and objects are created.

So typically you will see something like this:

[sourcecode language=”java”]
class Employee {
private int empId;
private String empName;

public int getEmpId(int aEmpId)
    {
           return empId;
    }
public int getEmpId(int aEmpId)
{
    empId = aEmpId;
}

public String getEmpName()
{
    return empName;
}

public void setEmpName(String aEmpName)
{
    empName = aEmpName;
}

}
class Contractor <strong>extends</strong> Employee {
}
class TestEmployee {
public static void main(String args[])
{
Employee emp1 = new Employee();
emp1.setEmpId(1);
emp1.setEmpName("John Doe");
Contractor cntr1 = new Contractor();
cntr1.setEmpId(2);
cntr1.setEmpName("Joe Contractor");
}
}
[/sourcecode]

In the above example, Employee is a typical Java class. In the main method, we are instantiating an object called, emp1 of type Employee. This allows us to access attributes and call methods in it. When you declare a class, you are actually inheriting from the Object class, which is the super class of all classes in Java.

In PB, we are dealing with such classes and Objects. For e.g., when you create a new Window in PB, you are actually creating such an inheritance. What is confusing in PB is that when you are inheriting from the standard (base) classes, you don’t use inheritance option. You actually use NEW option. This is often confusing to a newcomer and OO pundits.

PB came up with this design to make the OO concepts “hidden” to the newcomers. Several PB developers came from C and procedural languages as well as from Mainframe programming environment. Windows programming was new enough to a lot of them. Add OO concepts, it would be a longer learning curve. To ease this process, PB does a lot of work in the background.

When you create a new Control in PB, using NEW option, you are actually using inheritance. For e.g., when you create a new window and save it as w_employee, PB inherited from the base class called Window. This will be apparent if you “Edit Source” for the control. You will see the following line:

forward
global type w_employee from window
end type

Here w_employee is the new window we just created. Window is the base class we are inheriting from. “from”
is the keyword that points to inheritance. (In java it was “extends”).
